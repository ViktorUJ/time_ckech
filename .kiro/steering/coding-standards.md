# Стандарты кодирования

## Go

- Следовать стандартному форматированию `gofmt`
- Использовать `golangci-lint` для статического анализа
- Все публичные функции и типы должны иметь GoDoc-комментарии
- Обработка ошибок: всегда проверять и оборачивать ошибки через `fmt.Errorf("контекст: %w", err)`
- Не использовать `panic` в production-коде, кроме инициализации
- Контекст (`context.Context`) передавать первым параметром в функции с I/O

## Структура кода

- Бизнес-логика в `internal/` — не экспортируется наружу
- Точки входа в `cmd/` — минимальный код, только инициализация и запуск
- Интерфейсы определять рядом с потребителем, а не с реализацией
- Один файл — одна ответственность

## Windows-специфика

- Все вызовы Windows API оборачивать в Go-функции с понятными именами
- Пути к файлам: использовать `filepath.Join`, не хардкодить разделители
- Для работы со службами использовать `golang.org/x/sys/windows/svc`
- Для Event Log использовать `golang.org/x/sys/windows/svc/eventlog`
- Проверка системных процессов: по пути (`C:\Windows\`, `C:\Program Files\WindowsApps\`) и по цифровой подписи Microsoft

## Тестирование

- Unit-тесты для бизнес-логики (расписание, классификация процессов, подсчёт времени)
- Property-based тесты для критических инвариантов (корректность подсчёта времени, применение расписания)
- Моки для Windows API и HTTP-клиента в тестах
- Тесты запускать через `go test ./...`

## Логирование

- Все события записывать в Windows Event Log под источником "ParentalControlService"
- Уровни: Information (запуск/остановка), Warning (ошибки загрузки конфигурации), Error (критические сбои)
- Не логировать чувствительные данные (токены GitHub, пароли)
